# Архитектурный Реестр: Этап 0 (Завершено)

## 🏗 Система: Core Query Builder (@void/core-query-builder)
- **Логика:** Fluent API для генерации SQL (SELECT, INSERT, UPDATE, DELETE).
- **Безопасность:** Параметризация запросов ($1, $2) для защиты от SQL-инъекций.
- **Качество:** 100% TDD (8 тестов), нативный `node:test`.
- **Стэк:** Pure JS (ESM), JSDoc, Node.js 20.16.0.

## 🛠 Инфраструктура
- **Monorepo:** NPM Workspaces (связка пакетов и приложений).
- **Docker:** Единый Dockerfile для всей экосистемы, тесты проходят в контейнере.

## 🧠 Snapshot знаний (для интервью)
1. **Почему не ORM?** Контроль над SQL, понимание планов запросов (Explain), отсутствие лишних абстракций.
2. **Зачем плейсхолдеры?** Чтобы отделить структуру запроса от данных. Драйвер БД сам подставит значения, исключая возможность инъекции.
3. **Зачем Docker на этапе 0?** Гарантия того, что "работает на моей машине" = "работает везде".


# Архитектурный Реестр: Этап 1 (Auth Service — Логика)

## 🏗 Система: Auth Service (@void/auth-service)
- **Статус:** В процессе (Логика Domain и Use Cases завершена).
- **Domain Layer:**
    - Сущность `User`: инкапсулирует валидацию email (RegExp) и хранение хэша.
    - Инвариант: запрет на создание пользователя с некорректным форматом почты.
- **Use Case Layer:**
    - Сценарий `RegisterUser`: оркестратор регистрации.
    - Логика: хэширование -> создание сущности -> сохранение через репозиторий.
- **Infrastructure Layer:**
    - `UserRepository`: адаптер для `core-query-builder`. Выполняет маппинг полей (`camelCase` JS -> `snake_case` SQL).
    - `PasswordService`: обертка над библиотекой `argon2`. Использует алгоритм **Argon2id** (OWASP Standard).

## 🛠 Интеграция
- **Workspaces:** Пакет `@void/auth-service` успешно потребляет `@void/core-query-builder` через локальную связь NPM.
- **Безопасность:** Внедрено асинхронное хэширование. Пароли в открытом виде не покидают слой Use Case.

## 🧠 Snapshot знаний (для интервью)
1. **Почему Argon2id?** Это победитель Password Hashing Competition. Он лучше защищен от атак с использованием GPU (видеокарт) и ASIC по сравнению с BCrypt.
2. **В чем роль Use Case в Clean Architecture?** Это "дирижер". Он не знает, как работает база данных или как хэшируется пароль, он просто вызывает нужные методы в правильном порядке. Это позволяет менять библиотеку хэширования или базу данных, не трогая саму логику регистрации.
3. **Зачем маппинг в Репозитории?** Чтобы сохранить чистоту кода в JS (где принят camelCase) и соответствовать стандартам БД (где принят snake_case). Это разделение ответственности (Separation of Concerns).
