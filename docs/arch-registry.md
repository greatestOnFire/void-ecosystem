# Архитектурный Реестр: Этап 0 (Завершено)

## 🏗 Система: Core Query Builder (@void/core-query-builder)
- **Логика:** Fluent API для генерации SQL (SELECT, INSERT, UPDATE, DELETE, FOR UPDATE).
- **Безопасность:** Параметризация запросов ($1, $2) для защиты от SQL-инъекций.
- **Качество:** 100% TDD (9 тестов — добавлен тест блокировки), нативный `node:test`.
- **Стэк:** Pure JS (ESM), JSDoc, Node.js 20.16.0.

## 🛠 Инфраструктура
- **Monorepo:** NPM Workspaces (связка пакетов и приложений).
- **Docker:** Единый Dockerfile для всей экосистемы, тесты проходят в контейнере.

## 🧠 Snapshot знаний (для интервью)
1. **Почему не ORM?** Контроль над SQL, понимание планов запросов (Explain), отсутствие лишних абстракций.
2. **Зачем плейсхолдеры?** Чтобы отделить структуру запроса от данных. Драйвер БД сам подставит значения, исключая возможность инъекции.
3. **Зачем Docker на этапе 0?** Гарантия того, что "работает на моей машине" = "работает везде".

# Архитектурный Реестр: Этап 1 (Auth Service — Завершено)

## 🏗 Система: Auth Service (@void/auth-service)
- **Статус:** Завершено (Логика, Инфраструктура и HTTP API).
- **Domain Layer:**
  - Сущность `User`: инкапсулирует валидацию email (RegExp) и хранение хэша.
  - Инвариант: запрет на создание пользователя с некорректным форматом почты.
- **Use Case Layer:**
  - Сценарий `RegisterUser`: оркестратор регистрации (хэширование -> Entity -> Repo).
  - Сценарий `LoginUser`: аутентификация (поиск -> верификация -> генерация токенов).
- **Infrastructure Layer:**
  - `UserRepository`: Data Mapper для `core-query-builder`. Маппинг `camelCase` JS -> `snake_case` SQL.
  - `PasswordService`: Argon2id (OWASP Standard).
  - `TokenService`: Генерация и верификация JWT.
  - `SessionRepository`: Управление сессиями в Redis (TTL, Logout logic).
  - `Router`: Нативный HTTP-диспетчер с Dependency Injection.

## 🛠 Интеграция и Сеть
- **Docker Networking:** Сервис слушает `0.0.0.0`. Связь с БД и Redis через DNS-имена сервисов.
- **Security:** Реализован гибридный подход (Stateless JWT + Stateful Sessions в Redis).

## 🧠 Snapshot знаний (для интервью)
1. **Почему Argon2id?** Защита от GPU/ASIC атак, memory-hard алгоритм. Выбор для Senior-решений.
2. **В чем роль Use Case?** "Дирижер", который не зависит от деталей реализации (БД/Криптография).
3. **Зачем `0.0.0.0` в Docker?** Чтобы сервер принимал трафик с внешнего интерфейса контейнера, а не только внутри loopback.
4. **Зачем Redis для JWT?** Возможность мгновенного отзыва токена (Revocation), которую чистый JWT не обеспечивает.

- **Всего тестов:** 12 (Юнит, Интеграционные, Роутер).
- **Статус:** 100% Green.

# Архитектурный Реестр: Этап 1 (Payment Gateway — Завершено)

## 🏗 Система: Payment Gateway (@void/payment-gateway)
- **Статус:** Завершено (Логика, Блокировки, Идемпотентность, Распределенные события и HTTP API).
- **Domain Layer:**
  - Сущность `Wallet`: инкапсулирует данные баланса. Инвариант: баланс кошелька не может быть отрицательным.
  - Сущность `Transaction`: аудит движения средств. Инвариант: сумма транзакции (`amount`) должна быть строго больше нуля.
- **Use Case Layer:**
  - Сценарий `CreateWallet`: создание пустого кошелька для нового пользователя.
  - Сценарий `DepositFunds`: атомарное пополнение баланса в рамках ACID-транзакции.
  - Сценарий `TransferFunds`: перевод денег между кошельками с защитой от состояния гонки и дублей.
- **Infrastructure Layer:**
  - `WalletRepository` и `TransactionRepository`: адаптеры для генерации Pure SQL через `core-query-builder`.
  - `db.getTransactionClient()`: механизм выделения персонального клиента из пула для атомарных транзакций `BEGIN/COMMIT/ROLLBACK`.
  - `redis.get/set`: обеспечение идемпотентности через временные отпечатки ключей с TTL (24 часа).
  - `EventBus`: нативная шина распределенных событий на базе паттерна Redis Pub/Sub для интеграции Саги.
  - `Router`: нативный HTTP-диспетчер запросов.

## 🛠 Интеграция, Безопасность и Распределенная надежность
- **Row-Level Locking:** Использование `.forUpdate()` (`SELECT ... FOR UPDATE`) для пессимистического блокирования строк кошельков, исключающее потерю данных (Lost Update).
- **Exactly-Once Semantics:** Фильтрация повторных сетевых запросов через заголовок `x-idempotency-key` в Redis.
- **Saga Orchestration Ready:** Автоматическая публикация события `TRANSFER_COMPLETED` в топик `payment-events` для связи с внешними контекстами экосистемы.

## 🧠 Snapshot знаний (для интервью)
1. **Зачем вы добавили `FOR UPDATE` в свой Query Builder?** Для обеспечения строгой согласованности. Метод блокирует выбранную строку на чтение и запись для параллельных транзакций, предотвращая феномен *Lost Update* (потерянное обновление) при высокой конкурентности.
2. **Как вы решили проблему потери точности DECIMAL в Node.js?** Драйвер `node-postgres` возвращает тип `DECIMAL` в виде `String`. Я применил явное приведение к `Number` перед математическими операциями, чтобы избежать багов вычислений чисел с плавающей точкой.
3. **В чем разница между `db.query` и `getTransactionClient`?** Обычный `db.query` берет случайное соединение из пула на один запрос. Для транзакции ACID нужен один выделенный клиент, который последовательно выполнит `BEGIN`, серию запросов и `COMMIT` в рамках единого сетевого сокета.
4. **Как вы реализовали семантику Exactly-Once при сетевых лагах?** Использовал *Idempotency Key* в Redis. Если ключ найден в кэше — Use Case мгновенно возвращает старый сохраненный JSON-ответ, не запуская транзакцию в Postgres повторно.
