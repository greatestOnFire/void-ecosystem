# Архитектурный Реестр: Этап 0 (Завершено)

## 🏗 Система: Core Query Builder (@void/core-query-builder)
- **Логика:** Fluent API для генерации SQL (SELECT, INSERT, UPDATE, DELETE, FOR UPDATE).
- **Безопасность:** Параметризация запросов ($1, $2) для защиты от SQL-инъекций.
- **Качество:** 100% TDD (9 тестов — добавлен тест блокировки), нативный `node:test`.
- **Стэк:** Pure JS (ESM), JSDoc, Node.js 20.16.0.

## 🛠 Инфраструктура
- **Monorepo:** NPM Workspaces (связка пакетов и приложений).
- **Docker:** Единый Dockerfile для всей экосистемы, тесты проходят в контейнере.

## 🧠 Snapshot знаний (для интервью)
1. **Почему не ORM?** Контроль над SQL, понимание планов запросов (Explain), отсутствие лишних абстракций.
2. **Зачем плейсхолдеры?** Чтобы отделить структуру запроса от данных. Драйвер БД сам подставит значения, исключая возможность инъекции.
3. **Зачем Docker на этапе 0?** Гарантия того, что "работает на моей машине" = "работает везде".

# Архитектурный Реестр: Этап 1 (Auth Service — Завершено)

## 🏗 Система: Auth Service (@void/auth-service)
- **Статус:** Завершено (Логика, Инфраструктура и HTTP API).
- **Domain Layer:**
  - Сущность `User`: инкапсулирует валидацию email (RegExp) и хранение хэша.
  - Инвариант: запрет на создание пользователя с некорректным форматом почты.
- **Use Case Layer:**
  - Сценарий `RegisterUser`: оркестратор регистрации (хэширование -> Entity -> Repo).
  - Сценарий `LoginUser`: аутентификация (поиск -> верификация -> генерация токенов).
- **Infrastructure Layer:**
  - `UserRepository`: Data Mapper для `core-query-builder`. Маппинг `camelCase` JS -> `snake_case` SQL.
  - `PasswordService`: Argon2id (OWASP Standard).
  - `TokenService`: Генерация и верификация JWT.
  - `SessionRepository`: Управление сессиями в Redis (TTL, Logout logic).
  - `Router`: Нативный HTTP-диспетчер с Dependency Injection.

## 🛠 Интеграция и Сеть
- **Docker Networking:** Сервис слушает `0.0.0.0`. Связь с БД и Redis через DNS-имена сервисов.
- **Security:** Реализован гибридный подход (Stateless JWT + Stateful Sessions в Redis).

## 🧠 Snapshot знаний (для интервью)
1. **Почему Argon2id?** Защита от GPU/ASIC атак, memory-hard алгоритм. Выбор для Senior-решений.
2. **В чем роль Use Case?** "Дирижер", который не зависит от деталей реализации (БД/Криптография).
3. **Зачем `0.0.0.0` в Docker?** Чтобы server принимал трафик с внешнего интерфейса контейнера, а не только внутри loopback.
4. **Зачем Redis для JWT?** Возможность мгновенного отзыва токена (Revocation), которую чистый JWT не обеспечивает.

## ✅ Проверка качества (TDD)
- **Всего тестов:** 12 (Юнит, Интеграционные, Роутер).
- **Статус:** 100% Green.

# Архитектурный Реестр: Этап 1 (Payment Gateway — Завершено)

## 🏗 Система: Payment Gateway (@void/payment-gateway)
- **Статус:** Завершено (Логика, Блокировки, Идемпотентность, Фоновая Сага и HTTP API в рантайме Docker).
- **Domain Layer:**
  - Сущность `Wallet`: инкапсулирует данные баланса. Инвариант: баланс кошелька не может быть отрицательным.
  - Сущность `Transaction`: аудит движения средств. Инвариант: сумма транзакции (`amount`) должна быть строго больше нуля.
- **Use Case Layer:**
  - Сценарий `CreateWallet`: создание пустого кошелька для нового пользователя.
  - Сценарий `DepositFunds`: пополнение баланса в рамках атомарной ACID-транзакции.
  - Сценарий `TransferFunds`: перевод денег между кошельками с защитой от состояния гонки и дублей запросов.
- **Infrastructure Layer:**
  - `WalletRepository` и `TransactionRepository`: адаптеры для генерации Pure SQL через `core-query-builder`.
  - **Потокобезопасность (Thread Safety):** Использование паттерна фабрики `createQb = () => new QueryBuilder()` внутри `app.js` (Composition Root). Каждый вызов метода репозитория создает свой изолированный инстанс QB, что полностью ликвидировало проблему затирания контекста (`Table is not defined`) при параллельных HTTP-запросах.
  - `db.getTransactionClient()`: выделение персонального клиента из пула для атомарных транзакций `BEGIN/COMMIT/ROLLBACK`.
  - `redis.get/set`: обеспечение идемпотентности через временные отпечатки ключей с TTL (24 часа) для Exactly-Once семантики.
  - **SagaWorker:** Фоновый асинхронный процесс (демон). Использует выделенный дублирующий клиент `redisSub = redis.duplicate()` для прослушивания канала `payment-commands` через команду `SUBSCRIBE`. Автоматически перехватывает команды внешних сервисов и запускает юзкейсы переводов.
  - `Router`: нативный HTTP-диспетчер запросов со сквозным выравниванием DI-контекста.

## 🛠 Интеграция, Безопасность и Распределенная надежность
- **Row-Level Locking:** Использование `.forUpdate()` (`SELECT ... FOR UPDATE`) для пессимистического блокирования строк кошельков, исключающее потерю данных (Lost Update).
- **Exactly-Once Semantics:** Фильтрация повторных сетевых запросов через заголовок `x-idempotency-key` в Redis.
- **Saga Orchestration Ready:** Автоматическая публикация события `TRANSFER_COMPLETED` в топик `payment-events` через `EventBus.publish` сразу после успешного `COMMIT`.

## 🔒 Расширенная финтех-надежность (Дополнение)
- **Deadlock Prevention (Борьба со взаимными блокировками):** Внедрена детерминированная алфавитная сортировка UUID (`[id1, id2].sort()`) кошельков в рантайме Node.js перед вызовом `FOR UPDATE`. Это выстраивает параллельные транзакции в строгую очередь на уровне СУБД и исключает зависание процессов при встречных переводах.
- **Saga Compensation (Механизм отката):** Реализован изолированный сценарий `CancelTransfer` (Compensating Use Case) для асинхронного возврата средств (чарджбэка) со счета продавца на счет покупателя. Тип транзакции размечается для сквозного аудита с сохранением оригинального `reference_id`.
- **SagaWorker Evolution:** Фоновый демон успешно расширен для диспетчеризации и перехвата распределенной команды `CANCEL_TRANSFER` из канала `payment-commands`.

## 🧠 Snapshot знаний (для интервью на 3.5 млн)
1. **Зачем вы добавили `FOR UPDATE` в свой Query Builder?** Для обеспечения строгой согласованности. Метод блокирует выбранную строку на чтение и запись для параллельных транзакций, предотвращая феномен *Lost Update* (потерянное обновление) при высокой конкурентности.
2. **Как вы решили проблему потери точности DECIMAL в Node.js?** Драйвер `node-postgres` возвращает тип `DECIMAL` в виде `String`. Я применил явное приведение к `Number` перед математическими операциями, чтобы избежать багов вычислений чисел с плавающей точкой.
3. **В чем разница между вашим EventBus и SagaWorker в платежном шлюзе?** `EventBus` работает на отправку (Publish) — он выбрасывает событие `TRANSFER_COMPLETED` в Redis Pub/Sub сразу после успешного локального `COMMIT` в Postgres. `SagaWorker` — это фоновый слушатель (Subscriber). Он работает в параллельном потоке, слушает команды на списание средств из канала `payment-commands` и выступает триггером для запуска бизнес-логики шлюза со стороны внешних микросервисов.
4. **Зачем вы перешли на паттерн фабрики QueryBuilder в app.js?** Для обеспечения потокобезопасности (Thread Safety) в оперативной памяти Node.js. Изначально мы передавали один глобальный инстанс QB во все репозитории как синглтон. Под параллельной HTTP-нагрузкой возникло состояние гонки (Race Condition): один метод перезаписывал свойства таблицы или типа запроса другого метода до того, как тот успевал вызваться. Я изолировал контекст запросов, внедрив фабрику `() => new QueryBuilder()`.
5. **Как ваш платежный шлюз защищен от взаимных блокировок (Deadlocks)?**
   Если два пользователя одновременно переводят деньги друг другу, обычный `SELECT ... FOR UPDATE` заблокирует строку А в первом потоке и строку Б во втором, вызывая вечное ожидание (Deadlock). Я решил это внедрением лексикографической сортировки ID кошельков перед блокировкой. Все потоки всегда блокируют ресурсы в одном и том же порядке, выстраиваясь в безопасную очередь.
6. **Как вы обеспечиваете консистентность данных между микросервисами при сбое в распределенной системе?**
   Через паттерн Saga (Choreography/Orchestration). Если локальная ACID транзакция в Postgres платежного шлюза закрыта через `COMMIT`, её нельзя откатить обычным `ROLLBACK`. При сбое на следующем этапе (например, на Складе) во внешний топик Redis Pub/Sub публикуется команда `CANCEL_TRANSFER`. Наш фоновый демон `SagaWorker` перехватывает её и запускает компенсирующий Use Case `CancelTransfer`, который зеркально возвращает средства на баланс клиента и восстанавливает согласованность.
